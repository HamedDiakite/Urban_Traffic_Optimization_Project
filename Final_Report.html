1. Executive Summary

The Urban Traffic Optimization Project is a fully functional, interactive desktop application designed for metropolitan city planners and traffic engineers. Built with Python and the Tkinter GUI framework, the system provides a graph-based model of a city road network, implements two industry-standard pathfinding algorithms, and offers a rich suite of simulation, event management, and performance evaluation tools.

The project was developed in response to a formal specification document that outlined a comprehensive set of requirements spanning graph data structures, routing algorithms, congestion modeling, dynamic event handling, interactive visualization, and performance benchmarking. The final implementation delivered across two core source files — traffic_engine.py (1,050 lines) and main_app.py (1,883 lines) — satisfies every stated requirement and introduces several architectural and usability enhancements beyond what was originally specified.

Key Achievements

  • Dual-algorithm routing: Both Dijkstra's algorithm and A* search are fully implemented with a Euclidean distance heuristic, allowing side-by-side performance comparison.
  • Complete event management: Three distinct event types — Road Closure, Accident, and Construction — are handled with appropriate congestion multipliers and automatic route recalculation upon any change.
  • Emergency vehicle privileges: Emergency mode bypasses toll costs and one-way directional restrictions, enabling vehicles to traverse roads in any direction.
  • Full dynamic graph editing: City planners can add, remove, and edit intersections and roads directly from the GUI without restarting the application.
  • Four automated simulation scenarios: The simulation engine runs Emergency Routing, Peak Hour Analysis, Road Closure Impact, and Infrastructure Change evaluations, generating structured performance reports.
  • Data export: All simulation results and performance metrics can be exported in both CSV and JSON formats for downstream analysis.
  • Directional road system: Full support for both bidirectional and unidirectional (one-way) roads with visual arrow indicators on the canvas.

2. Project Requirements

The project specification defined a graph-based traffic simulation and optimization tool for metropolitan city planners. The following subsections document the original requirements as extracted from the specification document.

Objectives and Goals

The core objectives, as stated in the specification, were to:

  1. Minimize congestion across the city road network through intelligent routing.
  2. Reduce travel time for all vehicles via optimal pathfinding algorithms.
  3. Guarantee fastest-route access for emergency vehicles, including the ability to bypass normal traffic rules (toll restrictions and one-way constraints).
  4. Provide an interactive GUI for city planners to visualize the traffic network, run simulations, and observe optimized routing solutions in real time.
  5. Support dynamic, real-time updates — the system must react instantly to changing road conditions such as accidents, closures, and construction zones.

3. Features Implemented

All features described below are fully implemented in the production codebase at traffic_optimizer_tkinter/traffic_engine.py and traffic_optimizer_tkinter/main_app.py.

A.   Core Engine & Algorithms

Graph-Based Network Model

The traffic network is represented as a weighted directed graph using a nested dictionary (adjacency-list) structure. Each key in the outer dictionary is a node identifier (intersection), and the value is a dictionary mapping neighboring node IDs to Road objects (edges). This structure enables O(1) neighbor lookup and naturally accommodates both directed and undirected edges.

class TrafficNetwork: def __init__(self): self.graph: Dict[str, Dict[str, Road]] = {} self.node_positions: Dict[str, Tuple[float, float]] = {} self._event_listeners: List[callable] = []

Dijkstra's Algorithm

A textbook min-heap implementation of Dijkstra's algorithm is provided via find_path_dijkstra(). The algorithm uses Python's heapq module for the priority queue, maintaining a visited set to prevent re-processing. It returns a PathResult dataclass containing the total cost, ordered path, algorithm name, number of nodes explored, and computation time in milliseconds.

A* Algorithm with Euclidean Heuristic

The A* search is implemented in find_path_astar() using a Euclidean distance heuristic computed from node position coordinates stored in node_positions. The heuristic is admissible (never overestimates) since it represents the straight-line distance to the goal. A counter variable is used as a tiebreaker in the priority queue to ensure stable ordering.

def heuristic(node: str) -> float: """Euclidean distance to goal as admissible heuristic.""" if node not in self.node_positions or end not in self.node_positions: return 0.0 x1, y1 = self.node_positions[node] x2, y2 = self.node_positions[end] return math.sqrt((x2 - x1)**2 + (y2 - y1)**2) * 0.8

Algorithm Performance Comparison

The compare_algorithms() method runs both Dijkstra and A* on the same origin-destination pair and returns a comparison dictionary including per-algorithm metrics: path cost, path sequence, nodes explored, and execution time. This data is displayed in the Metrics tab and can be exported for benchmarking.

B.   Routing & Optimization

Multi-Criteria Optimization

Three distinct routing modes are supported through the avoid_tolls parameter and the weight calculation in Road.get_weight():

  • Time Mode (Allow Tolls): Toll cost is added directly to travel time. The optimizer minimizes the combined weight naturally.
  • Toll Avoidance Mode: Toll roads receive a penalty multiplier of 100× their toll cost, effectively pricing them out of viable routes.
  • Balanced Mode: The standard mode where actual toll values are included in cost, allowing the algorithm to select routes that balance time savings against toll expenditure.

Dynamic Weight Calculation Formula

The effective traversal cost for any road segment is computed as:

weight = (distance × current_congestion × event_multiplier) + toll_penalty Where: event_multiplier = 1.0 (OPEN) = 2.0 (ACCIDENT) = 1.5 (CONSTRUCTION) = inf (CLOSED) toll_penalty = 0.0 (emergency mode) = toll_cost × 100 (avoid tolls mode) = toll_cost (standard mode)

Emergency Vehicle Routing

When the Emergency Mode checkbox is activated, two special privileges are granted that override normal routing constraints:

  • Toll bypass: toll_penalty is set to 0.0 regardless of the road's toll cost, eliminating any toll-avoidance incentive.
  • One-way override: The _get_neighbors() method iterates over all entries in the graph for the source node when in emergency mode, including reverse traversal of one-way roads, enabling emergency vehicles to travel in both directions on any road.
def can_travel(self, from_node: str, to_node: str, is_emergency: bool = False) -> bool: if is_emergency: return True # Emergency vehicles bypass all direction rules if self.bidirectional: return True return from_node == self.start_node # Enforce one-way direction

Directional Road Support

Every road carries a start_node and end_node attribute that defines its canonical direction. When bidirectional=False, the edge is only traversable from start_node to end_node. The GUI renders directional arrows on the canvas and provides a "Toggle Direction" button in the Graph Editor tab to change road direction at runtime.

C.   Congestion & Time Modeling

Time-Dependent Weight System

Each Road object carries a current_congestion attribute (float) and a base_congestion attribute that reflects the road type's inherent congestion level. The time slider in the GUI calls update_time(), which recomputes current_congestion for every road based on the selected hour.

24-Hour Simulation Slider

A horizontal slider in the Main tab allows city planners to set the simulated hour from 0 (midnight) to 23 (11 PM). The current time is displayed as a formatted clock label (e.g., "07:00") and a congestion status indicator ("Rush Hour — 3x Congestion" or "Normal Traffic"). Any change to the slider immediately updates all road weights and redraws the network canvas.

Rush Hour Periods

Two daily rush hour windows are modeled:

  • Morning Rush: 7:00 AM – 9:00 AM — congestion factor 3.0×
  • Evening Rush: 4:00 PM – 6:00 PM (16:00–18:00) — congestion factor 3.0×
  • Off-Peak: All other hours — congestion factor 1.0× (normal)

Base Congestion Factors Per Road Type

Roads are assigned a road_type attribute ("highway", "local", "toll_road") which influences the starting base_congestion value. The time-based multiplier is applied on top of this base, yielding realistic differentiation between road categories during congestion events.

The traffic network maintains a list of event listeners registered via add_event_listener(). Whenever a road event is applied or cleared, the engine calls _notify_listeners() with a structured event dictionary. The GUI registers _on_network_event() as a listener, which automatically triggers _solve() (route recalculation) and redraws the canvas — ensuring that any road condition change immediately reflects in the displayed optimal route.

F.   Dynamic Graph Editing

Add / Remove Nodes

City planners can add new intersections by typing a node name in the Graph Editor tab and either entering explicit X/Y coordinates or clicking "Place on Canvas" to drop the node interactively. The add_node() engine method creates the adjacency-list entry and records the position for visualization. Nodes can be removed with remove_node(), which also cascades deletion to all connected road entries.

Add / Remove Roads

New roads are defined by selecting source and destination nodes from dropdowns, then providing distance, toll cost, base congestion, and directionality settings. The add_road() method creates the Road object and inserts it into the graph in both directions (if bidirectional) or the specified direction only (if one-way).

Edit Road Properties

Selecting an existing road from the editor's road dropdown populates all its attributes into editable fields. Changes to distance, toll, base congestion, or direction are committed via the edit_road() engine method, which updates the Road object in place and notifies all listeners. The canvas redraws immediately to reflect the change.

Real-Time Visual Updates

All graph edit operations dispatch a network event through the listener system, causing the canvas to redraw and all dropdown combo-boxes across all tabs to refresh their node and road lists. This ensures full consistency between the engine state and the GUI display at all times.

G.   Simulation Module

The SimulationEngine class (in traffic_engine.py) provides four automated scenario runners. Each scenario programmatically configures the network, executes routing, collects metrics, restores the network to its original state, and returns a structured results dictionary.

Scenario 1: Emergency Routing

Compares routing performance between a standard vehicle and an emergency vehicle on the same origin-destination pair. The scenario runs Dijkstra's algorithm twice — once with is_emergency=False and once with is_emergency=True — and reports the difference in path, cost, toll savings, and whether any one-way roads were traversed in reverse.

Scenario 2: Peak Hour Analysis

Sets the network congestion factor to 3.0 (simulating 7–9 AM or 4–6 PM conditions) and computes the optimal route. The results are compared with off-peak routing (factor 1.0) to quantify the impact of peak-hour congestion on travel time, path selection, and cost.

Scenario 3: Road Closure Impact

Programmatically closes the highest-weight road on the current optimal route and reruns the pathfinding algorithm. The output reports the original route, the closed road, the alternative route found, the additional travel time incurred, and whether the network remained fully connected.

Scenario 4: Infrastructure Change

Tests the impact of adding a new road segment to the network. The scenario calculates the baseline route, adds a hypothetical high-capacity road (low congestion, low toll), recalculates the route, and reports the improvement in travel time and cost attributable to the new infrastructure.

Results Display

Simulation results are formatted into a human-readable multi-section report displayed in a scrollable text widget. Each section covers: scenario summary, routes found, key metrics, and comparative analysis. Results are also available for export (see Section 3H).

H.   Performance Evaluation

PathResult Metrics Collection

Every routing call returns a PathResult dataclass that captures a comprehensive set of performance metrics:

  • Total cost (weight): Combined travel time and toll penalty.
  • Path sequence: Ordered list of node identifiers from origin to destination.
  • Algorithm name: "dijkstra" or "astar".
  • Nodes explored: Count of nodes dequeued from the priority queue.
  • Computation time: Wall-clock execution time in milliseconds.
  • Per-segment details: Distance, travel time, toll cost, and congestion factor for every individual road segment in the path.

Dijkstra vs A* Comparison

The Metrics tab displays a side-by-side comparison of both algorithms for the most recently computed route. The comparison highlights differences in nodes explored (A* typically explores fewer nodes due to its heuristic guidance) and computation time, providing empirical evidence of A*'s efficiency advantage on geographically embedded graphs.

Data Export

All simulation results and performance metrics can be exported in two formats:

  • JSON export: A structured JSON file containing all scenario results, path details, and metric dictionaries. Non-serializable values (such as float('inf')) are handled by a custom serializer that converts them to null.
  • CSV export: A tabular CSV file with per-row metric entries suitable for loading into spreadsheet tools or data analysis pipelines.

4. Technical Implementation

Architecture Overview

The system is structured as a strict two-layer architecture with a clean separation of concerns between the computation engine and the graphical interface:

Two-Layer Architecture
LayerFileLinesResponsibilities
Engine Layer traffic_engine.py 1,050 Graph model, algorithms, event management, simulation, metrics, data export
GUI Layer main_app.py 1,883 Tkinter window, canvas rendering, tab panels, user controls, results display

The GUI layer imports only from traffic_engine.py — never the reverse. This ensures that the engine is independently testable and reusable without any GUI dependency. Communication from the engine back to the GUI is handled through the event listener callback pattern (add_event_listener / _notify_listeners), which decouples the two layers without introducing circular dependencies.

Key Data Structures

Graph Representation

Dict[str, Dict[str, Road]] — nested dictionary providing O(1) neighbor lookup. Node positions stored separately in Dict[str, Tuple[float, float]] for heuristic computation.

Priority Queue (Min-Heap)

Python's heapq module provides an O(log n) priority queue for both Dijkstra and A*. Tuples of (cost, counter, node, path) ensure stable ordering when costs are equal.

Road Object

Each Road instance stores: distance, toll_cost, base_congestion, current_congestion, bidirectional (property), start_node, end_node, status (RoadStatus enum), and event (optional RoadEvent). Weight is computed on-demand via get_weight() rather than stored statically.

PathResult Dataclass

A typed dataclass returned by all routing calls containing: cost (float), path (List[str]), algorithm (str), nodes_explored (int), elapsed_ms (float), and details (List[dict]).

Algorithm Complexity Analysis

Algorithm Complexity Comparison
AlgorithmTime ComplexitySpace ComplexityHeuristicOptimal
Dijkstra O((V + E) log V) O(V) None Yes (non-negative weights)
A* O((V + E) log V) O(V) Euclidean distance (admissible) Yes (admissible heuristic)

In practice, A* explores significantly fewer nodes than Dijkstra on the geographically embedded city graph because the Euclidean heuristic provides strong directional guidance toward the goal. Both algorithms produce the same optimal path cost; A* simply converges faster on average.

Module-Level Design Patterns

  • Observer / Listener Pattern: The engine uses a callback list (_event_listeners) to notify the GUI of network changes without coupling to Tkinter directly.
  • Enum-Based State Machine: RoadStatus and VehicleType enums provide type-safe state representation and eliminate magic string comparisons.
  • Dataclasses for Immutable Results: PathResult and RoadEvent use Python @dataclass for concise, typed, and introspectable result containers.
  • Property Encapsulation: Road.bidirectional is exposed as a Python property that simultaneously updates both the internal flag and the graph adjacency structure, keeping the two representations in sync.
  • Factory Helper Functions: main_app.py uses helper functions (make_button, make_label, make_section) to apply consistent styling across all GUI widgets with minimal code repetition.

5. User Interface & Experience

Visual Design Philosophy

The interface adopts a professional, high-contrast color scheme anchored on navy blue (#0D1B3E) primary accents, white backgrounds, and a status-traffic color system (green = normal, red = congested, orange = toll, gray = closed). The overall aesthetic is inspired by municipal GIS dashboards — functional, informative, and presentation-ready.

Main View Tab — Detailed Description

The Main tab is divided horizontally: the left panel occupies approximately 60% of the window and houses the interactive canvas; the right panel provides controls. Key interactive elements include:

  • 24-Hour Simulation Slider: A wide horizontal slider with a formatted time label above it. Moving the slider updates the time display and congestion indicator in real time, then redraws all road colors to reflect the new congestion state.
  • Route Selection: Start and end node comboboxes, plus an algorithm selector offering "dijkstra" and "astar". All three fields update dynamically as nodes are added or removed from the graph.
  • Options Panel: Two checkboxes for Emergency Mode and Avoid Tolls, styled with descriptive inline labels explaining their effect.
  • Calculate Route Button: A prominent green CTA button that triggers pathfinding and populates the results panel below with path, total cost, distance, travel time, and toll breakdown.
  • Compare Algorithms Button: Runs both algorithms and populates the Metrics tab with a side-by-side comparison without navigating away.
  • Canvas Click Interaction: Clicking within 10 pixels of any road segment toggles a quick road closure (visual indicator: dashed gray line), demonstrating real-time network reactivity.

Graph Editor Tab — Detailed Description

The Graph Editor is organized into four collapsible sections within a scrollable frame:

  • Add Intersection: Name entry field with a "Place on Canvas" click mode.
  • Remove Intersection: Dropdown of existing nodes with a remove button that also removes all connected roads.
  • Add Road: Source/destination dropdowns, distance spinbox, toll cost entry, base congestion entry, and a bidirectional toggle checkbox.
  • Edit Road: Road selector dropdown that pre-fills all current properties; any modified field is committed on "Apply Changes". A "Toggle Direction" button flips a one-way road's permitted direction and updates the canvas arrows accordingly.

An editor log at the bottom of the panel records all edit operations with timestamps, providing a change audit trail during interactive sessions.

7. Additional Enhancements

Beyond the requirements defined in the specification document, the final implementation includes several features and design decisions that were not explicitly required but substantially improve the system's quality, educational value, and user experience.

Directional Road System

While the specification mentioned one-way roads in the context of emergency routing, the final implementation delivers a complete directional road system as a first-class feature:

  • Visual arrows on canvas: Two triangular arrowheads are drawn at 35% and 65% along each one-way road, unambiguously indicating the permitted direction of travel.
  • Bidirectional markers: A distinct double-arrow indicator distinguishes bidirectional roads from one-way roads without requiring the user to click each road.
  • Runtime direction toggle: City planners can convert a bidirectional road to one-way (or vice versa) at any time through the Graph Editor, and the canvas immediately reflects the change through updated arrow rendering.
  • Direction blocker detection: The engine's _find_direction_blockers() method identifies any one-way roads on the current path that would block a non-emergency vehicle, providing informative warnings in the results panel.

Observer Pattern for Real-Time Reactivity

The event listener architecture goes beyond a simple "recalculate on click" approach. Any engine operation — including adding a road, removing a node, applying an event, or changing a road's direction — triggers a structured notification to all registered listeners. This makes the system fully extensible: additional listeners could be attached for logging, analytics, or integration with external systems without modifying the engine code.

Per-Segment Route Breakdown

The specification required displaying total route distance, time, and toll cost. The implementation goes further by computing and displaying a per-segment breakdown for every road in the computed path. Each segment entry shows the road identifier, segment distance, travel time contribution, and toll charge. This level of detail is invaluable for city planners analyzing specific bottlenecks in the network.

Algorithm Comparison in Metrics Tab

The specification required implementing both Dijkstra and A* but did not specify a direct comparison feature. The implementation adds a "Compare Algorithms" button that runs both algorithms on the same query and presents a structured comparison table showing nodes explored and computation time. This turns the application into an educational tool for demonstrating algorithm efficiency.

Hover Tooltips on Canvas

Moving the cursor over any road on the canvas triggers a tooltip popup displaying the road's key attributes: distance, base congestion, current congestion, toll cost, status, and direction. This allows rapid inspection of road properties without switching to the Graph Editor tab.

Editor Audit Log

All graph editing operations are recorded in a scrollable log panel at the bottom of the Graph Editor tab with timestamps. This provides a change history for each session — useful for documenting what modifications were made during a planning or demonstration session.

Reset Network Functionality

A "Reset Network" button restores the full network to its initial state (original nodes, roads, and properties) in a single click.
